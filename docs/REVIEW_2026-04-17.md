## TL;DR

You've built more than you remember. The core pipeline (Wanted → Search → Download → Organize → Library) works end-to-end and 5 of the 6 imagined ingestion sources are already wired (Spotify, YouTube, SoundCloud, Discogs, Bandcamp). What's blocking the dream isn't features — it's **glue, polish, and trust**:

1. **Background jobs die on every container restart.** Today's session lost a 5 GB download because of this. The cleanup hook I added (`backend/traxdb/apps.py`) is a band-aid; the real fix is a separate worker container (Celery / Dramatiq / Huey) or graceful-shutdown semantics. Until this is fixed nothing here is "automation," it's "babysat scripts."
2. **The TraxDB → Organize bridge does not exist.** Files land in `/music/Electronic/ID3/traxdb/YYYY-MM-DD/` and are invisible to the rest of the pipeline. Every other module integrates; TraxDB is a satellite.
3. **The Recognize module silently misidentifies tracks** — clustering bugs, no manual override, no live progress, can't trust on a 2 hr mix. Agent 24 calls confidence ~70%. This is the thing the user *most needs to work*; right now it's the thing most likely to mislead.
4. **No tests anywhere.** With 7 distinct status enums, threading-everywhere, and cross-module FKs, this is a fragile composition. One unit test on the search→download→organize state transitions would catch half of today's session bugs.
5. **The vision differentiator (auto-classify into your own crates via audio embeddings) is a weekend's work** on top of the existing library module. Stack: Essentia + Discogs-EffNet + scikit-learn. No GPU needed. Real moat vs. every commercial tool.

If I had to rank what to do next, in order:
1. Fix recognize module bugs + add manual track override (it's the marquee differentiator that's broken).
2. Bridge TraxDB → Organize so all downloads converge (kills a major UX break).
3. Build the SourceAdapter abstraction so adding Shazam / Safari Tab Group / SoundCloud Likes is a 50-line file each.
4. Move long-running jobs to a worker container (or accept threads + graceful shutdown).
5. Ship the v0 auto-classifier as a personal-use proof — it's a unique-in-the-market feature.

Everything else is polish.

## Vision recap — mirror what I understand

You want OCDJ to become the brain that watches every place you save tracks (Spotify playlists, SoundCloud likes, YouTube playlists, Shazam history, Discogs wantlist, a Safari tab group full of Bandcamp/blog pages, plus the browser extension you already shipped) and routinely:

1. Pulls those new "wants" into a central queue.
2. Recognizes the actual track (especially for mixtape URLs where you only have audio).
3. Finds it on Soulseek at the highest quality you'll accept.
4. Downloads it, tags it, renames it, converts it to your house format (AIFF), files it.
5. Optionally pushes it into iTunes / Music.app / Rekordbox.
6. (Long-term) Looks at the spectrogram + your existing crates, predicts which crate the new track belongs in, surfaces top-3 with confidence, learns from your corrections.

Initially personal. Possibly a product later.

**Where the codebase aligns:** all 4 stages (capture → curate → fetch → process) exist as Django apps with working models and APIs. Auto-pipeline orchestrator (`backend/core/services/automation.py`) already coordinates search→download→organize behind 3 toggles. Frontend has a panel per stage. The architectural skeleton is sound.

**Where it diverges:**
- "Routinely scan" — there is **no scheduler**. Every ingestion is manual. APScheduler / cron / Celery beat is missing.
- Several promised sources are not yet wired (Shazam, Safari Tab Group, YouTube Watch Later, SoundCloud Likes — only playlists/sets).
- TraxDB downloads do not feed the rest of the pipeline — files sit in their own root.
- iTunes integration: zero. Only an Apple Music URL field on Shazam recognitions.
- Auto-classify: not started.

## Architecture findings

### Threading model is the #1 architectural problem

**Map:** 13+ sites spawn raw `threading.Thread(daemon=True)` across `soulseek`, `recognize`, `organize`, `library`, `traxdb`, `wanted`, `core/automation`. Every one of them dies on container restart.

**Today's evidence:** I restarted `ocdj-backend-1` six times during the bug-fix session. Each restart killed the in-flight 14 GB TraxDB download. The cleanup hook I added catches the orphaned `running` row on next boot, but the work itself is gone — only saved by the file-level downloader being idempotent.

**Three options to consider:**
- **Stay on threads, add discipline.** Keep things simple. Add: SIGTERM handler that waits up to N seconds for threads to finish, the cleanup hook (already done for traxdb), health endpoint that reports running threads, "resume from last checkpoint" semantics for every long worker (recognize already has it; traxdb needs it; organize doesn't need it because work is per-file).
- **Adopt Dramatiq or Huey** (lighter than Celery, no broker required for Huey on SQLite/Redis). Worker runs in a separate container, jobs persist as DB rows, retries are automatic.
- **Adopt Celery + Redis + Flower.** Heavier, but mature, observable, well-known. Probably overkill for a 1-user app, becomes essential if you release publicly.

My pick for *now*: option 1 + add resume to TraxDB. For *if you ship it publicly*: option 2 (Huey).

### State machine sprawl

7 separate status/stage enums across models, with overlapping vocabulary ("downloading" means three different things). No single "track journey" object you can point at and say "this is where Hanna - Punk is in the pipeline."

**Recommendation:** an optional `AutomationTask` table that owns the canonical state and points to the underlying Wanted/SearchQueueItem/Download/PipelineItem rows. Don't replace the existing models — reframe the existing rows as *operational records* and add a single record-of-truth on top.

This also fixes the UX problem (see UX section): the user could open a single row in Wanted and see the entire downstream history without panel-hopping.

### Configuration sprawl

Three sources of truth: `.env`, `settings.py`, `core/Config` DB table, plus per-module constants in code. Today's Pixeldrain key change was easy because it lives in `.env`; the conversion rules live in DB; the recognize segment lengths live in module constants. No discoverability.

**Recommendation:** central `get_config(key)` helper that resolves DB → env → settings → schema default in that order, with a registry that lists every key. Then a single `/settings` URL that shows the full config and lets you override what's overrideable.

### Cross-module coupling

- **Wanted ↔ Soulseek** is well-coupled via FKs and explicit status sync code in `soulseek/views.py:336-345` (now atomic, fixed today).
- **Soulseek → Organize** auto-handoff via `auto_ingest_download()` (now correct after today's `transaction.on_commit` fix).
- **Recognize → Wanted** via `add_to_wanted()` — works but creates a new "Recognize" source per job rather than preserving the originating mix URL. Lose-able provenance.
- **TraxDB → anything** = nothing. Zero coupling. The biggest integration debt.
- **Library** is a fully read-only mtime walk over `05_ready/`. Doesn't index anything outside that folder.

### Test coverage = 0

No `test_*.py`. No CI. Top 5 to add first per the architecture agent:
1. Automation cycle FSM (state transitions, no orphaned items)
2. Soulseek score + match (catalog # boosts, query simplification)
3. Organize pipeline stage movement (file discovery, collisions)
4. Recognize segmentation + cluster merging
5. Cross-module state sync (Wanted ↔ SearchQueueItem ↔ Download ↔ PipelineItem)

## Recognize module deep dive

This is where the most user pain lives, and the most differentiation potential.

### What works
- Multi-engine pipeline: ACRCloud (primary, 20s step) → Shazam (gap fill, 8s step) → TrackID.net merge.
- Resume after restart (raw_results saved per pass).
- Cluster merging, conflict resolution, ACRCloud noise filter.
- Description parser pulls embedded tracklists from YouTube/SC for cross-validation.

### Concrete bugs (P0 — fix this week)

1. **`asyncio.run()` reuse crash** in `recognition.py:122` — second call in same thread blows up. Affects gap-fill after primary pass.
2. **Gap-fill dedup missing** in `pipeline.py:239` — `raw_results.extend(gap_results)` then resort, no overlap check. Same track counted twice → inflated `segment_count` → bad clustering.
3. **TrackID Cloudflare bypass stale** — hardcoded `cf_clearance` cookie eventually breaks silently.
4. **Resume path** — if audio file exists but `raw_results` empty/corrupt, skips primary pass and uses empty results. Should validate consistency.
5. **Worker thread crash leaves job stuck `recognizing` forever** with no retry path. (Same root cause as the threading model issue.)

### Bugs in clustering / output (P1)

- Overlap calc in `_merge_trackid_results` uses `tt_duration * 0.5` as threshold — wrong for short DJ-overlaps (10–30s).
- Conflict window of 30s in `clustering.py:353` rejects legitimate mashups (two tracks playing simultaneously).
- ACRCloud region default `eu-west-1` — silent zero-results if creds are for a different region.

### Quality vs commercial tools

| | OCDJ today | TrackSniff (commercial) | DJ.Studio | Shazam alone |
|---|---|---|---|---|
| Engines | 3 | 2 | 2 | 1 |
| Underground coverage | Excellent | Good | Good | Poor |
| Live progress | No | Yes | Yes | Yes |
| Manual edit/correction | No | Yes | Yes | Yes |
| Restart-safe | Yes | n/a | n/a | n/a |
| ~Reliability on 2hr mix | ~70% | ~85% | ~85% | ~40% |

### Recommendations (priority order)
1. Fix the 5 P0s above. ~1-2 days.
2. Add per-row "+ Wanted" + bulk "add all verified" buttons (UX fix #1).
3. Add manual edit endpoint: `PATCH /recognize/jobs/<id>/track/<idx>/` to override artist/title and mark `verified=true`. Re-runs leave verified rows alone.
4. Add fingerprint cache: `(audio_hash, segment_index) → result`. Stops re-querying ACRCloud for the same mix.
5. Add SSE / WebSocket live progress. Right now it polls; users can't see partial results stream in.
6. Hybrid confidence rerank: cross-engine validation should boost score (matched-by-2 = high confidence even if Shazam score is mid).

If you ship publicly, this is the marquee feature. Make it bulletproof.

### Architectural recommendation: SourceAdapter

Rather than 7 bespoke service files in `wanted/services/`, formalize a base class:

```python
class TrackSource(ABC):
    source_type: str
    display_name: str

    @abstractmethod
    def authenticate(self, params) -> AuthToken: ...

    @abstractmethod
    def fetch_playlist(self, playlist_id, token) -> List[TrackMetadata]: ...

    @abstractmethod
    def list_playlists(self, token) -> List[Playlist]: ...

    def deduplicate(self, tracks): ...  # default uses dedup.py
```

Existing Spotify adapter already follows this pattern informally (`services/spotify.py:145-165`). Convert it to the base class, then porting the 4 missing sources is mechanical.

### Scheduler

There's no polling loop. Add `poll_interval_hours` and `next_sync_at` to `WantedSource`, create a management command `sync_wanted_sources` that iterates due sources, run it from cron or APScheduler. Show "next sync in 14 min" in the UI per source.

### Cross-source dedup

`dedup.py` checks within an import operation only. A track saved on both Spotify and Shazam creates two `WantedItem`s today. Add cross-source merging via either:
- A `fuzzy_signature` column with hash collision detection, or
- A `merge_into_id` self-FK with manual + auto-merge endpoints, or
- A `source_ids = JSONField()` to record "this track was found in: spotify, shazam".

## ML auto-classification feasibility

**This is doable as a weekend project.** No GPU required.

### Recommended stack (April 2026)

- **Feature extraction:** [Essentia](https://essentia.upf.edu/) (C++ kernels with Python binding, ~2x faster than librosa, ships TensorFlow model zoo).
- **Embedding model:** **Discogs-EffNet** (5M params, 1280-dim embedding, trained contrastively on 2M+ Discogs tracks for artist/label/release similarity — almost exactly "what makes a track belong in playlist X").
- **Classifier:** scikit-learn `LogisticRegression(class_weight='balanced')` per playlist.
- **Storage:** SQLite vector cache (5000 tracks × float32[1280] ≈ 25 MB). Don't write embeddings to ID3.

### Why this is a real moat

Every commercial DJ tool gives you *their* taxonomy:
- Cyanite.ai — generic "house/techno" tags, €290/mo+
- Mixed In Key — key/cue/energy only, no classification
- Rekordbox 7 — auto-playlists by metadata matching, not audio similarity
- Djoid — €149+€99/yr, sonic graph but their categories
- Lexicon DJ — cross-app library sync, no classification

**No tool lets you point at "my Sunday Warm-Up" crate and say "learn that."** That's the OCDJ differentiator.

### Hardware reality check (5000 tracks on M-series MacBook)

- Discogs-EffNet embedding extraction: ~2s/track CPU → 5000 tracks ≈ **~3 hrs one-time**
- Classification of a new track: extract one embedding (2s) + LR predict (<10ms) → real-time
- RAM peak: <2 GB
- TensorFlow on Apple Silicon: use `tensorflow-metal` for ~3x speedup, or accept CPU.

### Few-shot reality

You'll have 50–200 tracks per playlist. That's plenty for a linear classifier on top of a frozen embedding — the standard MIR baseline that consistently within 1–2% of fine-tuned models on small-n. No deep-learning fine-tuning needed.

### v0 design (~200 lines Python)

```
poc/
├── extract.py    # Essentia MonoLoader + TensorflowPredictEffnetDiscogs
├── train.py      # Per playlist: LogisticRegression on stacked embeddings
├── classify.py   # Predict_proba ranked top-3 with confidences
└── run.py        # CLI
```

Expected accuracy: >95% on clean distinctions (Deep House vs DnB), 75–85% on subtle (Peak Time vs Melodic Tech House — the latter is also where humans disagree).

### Workflow recommendation

- Bootstrap from existing iTunes / Rekordbox XML export (read folder names as labels).
- Classify-on-import: new track → top-3 with confidence bars + "which crate?" UI.
- Active learning v2: when batch-importing, sort by classifier *uncertainty*, ask user to label highest-value tracks first.

### Where it fits in OCDJ

Add a new `classify` Django app:
- `Embedding` model (track FK, vector, model_version)
- `PlaylistClassifier` model (name, scaler, model bytes)
- `Classification` model (track FK, playlist FK, confidence, manual_override, accepted_by_user)
- Service: `extract_embeddings(track_ids)`, `train_classifier(playlist_id)`, `predict(track_id)`

Tie into Library: every `LibraryTrack` gets an embedding. Tie into Wanted: when a track lands in `05_ready`, run classification, surface top-3 in a new "Triage" view.

## Competitive landscape

### The closest competitor is not a product, it's a stack

**Lidarr + slskd + Soularr** is what the self-hosted music community has been gluing together for 2–3 years. OCDJ's killer differentiator vs that stack: **track-first**. Lidarr forces full albums; OCDJ is track-native from day one. This is Lidarr's #1 documented UX flaw and the single biggest reason DJs don't use it.

### Direct DJ-targeted tools

| Tool | Pricing | What it does | OCDJ overlap |
|---|---|---|---|
| TrackSniff | $9–19/mo, 50 free/mo | Mixtape URL → tracklist with BPM/key/genre | Recognize module overlap |
| Djoid | €149 + €99/yr | AI tagging (BPM/key/energy/emotion), Chapter Builder | Library/classify overlap |
| MusicMate | n/a | AI track discovery from Beatport/SC/Bandcamp | Wanted overlap |
| Lexicon DJ | $79–399 lifetime | Cross-app library manager (Rekordbox/Serato/Traktor sync) | Library/Organize overlap |
| Mixed In Key | $79.99 perpetual / $7.99 mo | Key, cues, energy | Adjacent — integrate, don't replace |
| Cyanite.ai | €290+/mo API | Auto-tagging (1500+ keywords) | Classify overlap (commercial alt) |

### Open source to learn from

- **Lidarr** — file naming templates `{Artist}/{Album} ({Year})/{track:00} - {Title}` are worth copying.
- **beets** — 150+ plugin ecosystem, killer query language `beet list 'love bpm:120..130 key:c#m'`. Study the plugin pattern.
- **slskd** — already integrated.
- **Picard** — AcoustID fingerprinting pipeline; integrate, don't reinvent.
- **sldl / slsk-batchdl** — closest CLI analog to one of OCDJ's loops.
- **spotDL** — Spotify URL → YouTube audio → tagged file.

### The actual moat

**Two of three differentiators already exist as point tools; the composition is the moat.** No one else does mixtape-ID → Soulseek → classify-into-your-crates as one unattended pipeline. TrackSniff stops at tracklist. Soularr starts from Lidarr wantlist. sldl accepts CSV but doesn't *produce* one from a SoundCloud mix.

**Five things no current product does:**
1. End-to-end mixtape → tracklist → Soulseek → tagged + sorted into a working DJ library, unattended.
2. DJ-centric source fanout (Spotify + SC + YT + Shazam + Discogs unified).
3. Album-skipping, track-first model.
4. Per-DJ auto-classification into YOUR existing playlist taxonomy (vs generic genre tags).
5. Self-hosted, BYO-Soulseek, no label deal needed.

### Pricing sweet spot if you ship

**Free OSS core + $79 one-time Pro license** (adds auto-classify, set-builder, priority indexers). Avoid subscription for self-hosted — that audience revolts. Hosted tier $7/mo for non-technical DJs is plausible but legally hot lava (BYO-Soulseek is mandatory; never host the Soulseek credential).

### Honest ceiling

Realistic: low thousands of paying users at $79–149. Lexicon-for-sourcing scale, not VC. Pulselocker died trying to license DJ streaming; anything above ~$10M revenue requires label deals which are prohibitive for a solo dev.

### Distribution

Launch on **DJ TechTools + r/DJs + r/selfhosted + Hacker News**, not Product Hunt. The audience is 50% DJs who want better workflow, 50% self-hosters who love the *arr stack — and nobody currently serves both well. Naviseerr tried and stalled; the window is open.

### Legal posture (copy from Lidarr)

> OCDJ is a tool for organizing your personal music collection. You supply your own sources and accounts. Users are responsible for compliance with the TOS of any third-party service they connect, and with copyright law in their jurisdiction.

BYO Soulseek account, BYO API keys, fingerprint-then-discard mix audio (don't persist rips), self-hosted core. Same posture as Lidarr/beets/Soularr.

## Cross-cutting recommendations

### 1. Promote `core/automation.py` to a real automation engine

Today: 3 toggle bits + `run_automation_cycle()` function. To support the vision, it needs:
- Scheduler (APScheduler embedded or external cron) — pull from sources every N hours, run automation cycle every N minutes.
- Per-step locks (already have `_processing_all_lock` pattern in organize) so concurrent triggers don't race.
- Audit log (`AutomationRun` model with start/end/items_processed).
- Backpressure: don't spawn unlimited search threads when 100 wanted items go pending at once.

### 2. Build a real `AutomationTask` model on top of existing models

Don't replace WantedItem etc. Add a parallel record-of-truth that captures the journey:

```python
class AutomationTask(models.Model):
    wanted_item = models.OneToOneField('wanted.WantedItem')
    queue_item = models.ForeignKey('soulseek.SearchQueueItem', null=True)
    download = models.ForeignKey('soulseek.Download', null=True)
    pipeline_item = models.ForeignKey('organize.PipelineItem', null=True)
    library_track = models.ForeignKey('library.LibraryTrack', null=True)
    state_history = models.JSONField(default=list)  # auditable
    error_message = models.TextField(blank=True)
```

Frontend opens a Wanted row → sees the entire downstream history without panel-hopping.

### 3. Bridge TraxDB → Organize

Today TraxDB downloads to `/music/Electronic/ID3/traxdb/2026-04-13/` and the rest of the pipeline ignores them. Two options:
- **Auto-create PipelineItems** for each downloaded ScrapedTrack, route through organize like Soulseek downloads.
- **Treat traxdb as its own ready-zone** parallel to Library — let user "promote to library" per release.

The first is purer; the second matches the user's mental model of "TraxDB = their archive, separate from Soulseek discoveries." Pick one and commit.

### 4. Configuration consolidation

`get_config(key, default=None, cast=str)` resolving DB → env → settings → schema default. Every module imports from one place. Frontend reads/writes to one settings page that lists every key.

### 5. Test coverage minimum

Five tests this month:
1. `test_automation_cycle.py` — full pipeline FSM with mocked SlskdClient
2. `test_soulseek_scoring.py` — score_result on known cases
3. `test_organize_pipeline.py` — file discovery + stage movement
4. `test_recognize_clustering.py` — segment merging + dedup
5. `test_state_sync.py` — Wanted ↔ SearchQueueItem ↔ Download status sync

Add `pytest` + `pytest-django`, run on commit. Even 5 tests catch regressions.

### 6. Background jobs (pick one)

- **Now:** stay with threads + add SIGTERM grace + cleanup hook (already done for traxdb; do for soulseek/recognize/organize).
- **If you ship publicly:** Huey (lightweight, persistence via Redis or SQLite, no broker setup) or Dramatiq (Redis-based, Celery-like API but simpler).

### 7. Library module needs a face-lift

Current state: bare metadata browser. Needs (in order):
1. Album artwork extraction + display.

## Open questions for you

1. **Personal-use or product?** Both are valid. Personal = optimize for *you*, ship features that scratch your itch first, screw the moat analysis. Product = polish + abstraction + onboarding from day one. Knowing this changes the next 3 months of priorities.

2. **TraxDB integration model:** auto-pipeline through organize like Soulseek downloads, or keep as separate archive that you promote-to-library manually? Cleaner data flow vs matching how you actually think about it.

3. **iTunes integration:** is it just "drop file into Music.app" (AppleScript, easy) or something deeper like syncing playlists, ratings, play counts? Big spread in scope.

4. **Recognize trust:** are you willing to invest in fixing it properly (manual override, fingerprint cache, live progress) or do you treat it as best-effort and always sanity-check yourself? Determines how much engineering goes there.

5. **Auto-classifier vision:** classify-on-import only (suggest top-3, you accept) or fully unattended (just files itself)? The former is high-trust tool, the latter is risky and rarely what people actually want.

6. **Public release timeline:** if "yes someday," what's "someday"? 6 months? 12? Defines whether to spend time on test coverage / docs / onboarding now.

7. **Background jobs:** is "don't restart while a long job runs" forever an OK constraint, or do you want jobs that survive restarts? The latter forces a worker container.
